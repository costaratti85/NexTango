## MSG_022 — Fix "No se detectaron figuras" (vectorizador v2)

**Causa raíz confirmada:** el parser de potrace SVG en `runner.py` tenía
`if d and "z" in d.lower()` — filtraba paths sin `Z` explícito. Para una
retícula conectada (un solo componente grande), potrace puede trazar el
contorno externo de una forma que en edge cases carecía de Z → filtrado → 0 entidades.

**Fix:** quitado el filtro. Ahora `if d:` — si potrace trazó algo, se muestra.
La arquitectura v2 (presets × entities, sin matching de "figuras") ya era la
solución estructural; este fix es el parche defensivo complementario.

**Nota para Vega:** el campo `figuras` desapareció en v2. Si el frontend de
`vectorizar_patron` todavía lee `r.message.figuras`, necesita actualizar a
`r.message.presets[i].entities`. Dejé el contrato en MSG_019 del canal de Vega.

## MSG_020 — Thumbnail automático

**`_generate_and_save_thumbnail(nombre, dxf_path)`** (privado, en `patrones.py`):
- ezdxf + matplotlib Agg (headless) → PNG 216×216px en
  `public/pattern_thumbnails/{safe_name}.png`
- Silencioso si falla (matplotlib ausente, DXF inválido) — nunca rompe el upload

**Enganchado en:**
- `upload_pattern` → genera thumbnail del DXF recién copiado a /planos/
- `compose_pattern` (vectorizer) → genera thumbnail del DXF compuesto

**`backfill_thumbnails()`** (whitelisted) para Orbit:
```bash
bench --site erp.local execute sistema_industrial.api.patrones.backfill_thumbnails
```
Recorre SI Patron activos tipo Archivo/Vectorizado sin PNG en disco, genera.
Retorna `{generados, ya_existian, errores}`.

**Fix case-sensitivity:** PNG siempre se guarda como `{_safe_name(nombre)}.png`
(ej. `Subte.png`) — consistente con lo que busca `_thumbnail_url()`. El problema
de "subte" vs "Subte.png" se resuelve sólo al correr el backfill.

**Prerequisito:** `ezdxf[draw]` debe estar instalado en el venv de Frappe.
Si no lo está, el render falla silenciosamente — los patrones suben igual, sin thumbnail.
Forge puede verificar con `bench pip show ezdxf`.
